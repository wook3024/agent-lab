Project Snapshot

최고 품질을 위한 에이전트 운영 체계를 실험하고, complex-case에서 실제로 이기는 조합을 골라낸 프로젝트

이 프로젝트는 간단한 답변 품질이 아니라, 여러 파일에 걸친 구현, 회귀 위험 버그 수정, 테스트/문서 동기화, 운영 리스크 통제 같은 복잡한 제품 개발 작업에서 어떤 agent + skill + model + reasoning effort 조합이 가장 안정적인지 검증하기 위해 설계되었습니다.

Current verdict broad sweep에서는 c2-mini-triage-context가 가장 효율적이었고, refined confirmatory batch까지 포함한 현재의 보수적 기본값은 C2 + all gpt-5.4/high입니다.

Best Validated Default

C2 + all gpt-5.4/high

품질 보장 최우선 기준의 기본 승격 조합. refined batch에서 0 finding을 기록했습니다.

운영 가설과 판정 기준

이 프로젝트가 “좋은 프롬프트”가 아니라 “품질을 보장하는 운영 시스템”을 만들기 위해 잡은 핵심 철학입니다.

single specialist first independent review required reasoning effort is a knob complex-case over simple tasks

출발점은 복잡한 개발 작업에서 무너지는 지점이 코드 생성 자체보다 범위 통제, 컨텍스트 선택, 회귀 검증, 문서 동기화, 독립 리뷰에 더 많이 있다는 가정입니다. 그래서 baseline은 멀티에이전트 확장이 아니라 Triage → Context → Execution → Gates → Review 형태의 운영 파이프라인으로 설정했습니다.

  • simple task 최적화는 비목표입니다.
  • 속도보다 요구사항 충족, 회귀 방지, 테스트 보강, 문서 반영을 우선합니다.
  • trace와 deterministic gate 없이 “좋아 보이는” 조합은 승격하지 않습니다.
  • Primary metrics: success rate, gate pass rate, High finding rate, docs sync success.
  • Secondary metrics: latency, token cost, human intervention frequency, taxonomy drift.
  • Failure tags: wrong_context, over_edit, missed_test, docs_desync, shallow_fix.
  • Complex-case 기준: 3개 이상 파일, 테스트 필수, 상태 전이/정책/문서 반영/회귀 위험 중 3개 이상 충족.

후보 에이전트 Topology

C0와 C2를 실제 benchmark 대상으로 두고, single-pass와 controlled dual-pass가 complex-case에서 어떻게 갈리는지 비교했습니다.

Baseline single-pass

C0 Codex Solo

하나의 에이전트가 task brief, context pack, 구현, 테스트, 문서 반영까지 모두 한 번에 처리하는 비교 기준점입니다. 구조는 단순하지만 범위 통제와 자기검열이 약해질 가능성이 큽니다.

Triage + Context 단일 실행 안에서 브리프와 컨텍스트를 직접 작성
Execution 코드 변경, 테스트, 문서 갱신을 한 번에 수행
Gates 결정적 검증을 바깥에서 실행
Review 후행 pass로만 문제를 발견
  • 장점: 운영 오버헤드가 낮고 handoff failure가 없습니다.
  • 약점: 잘못된 초기 가정을 스스로 교정하기 어렵고, missed test와 over-edit에 취약합니다.
  • 최종 판정: baseline reference로만 유지, 기본 조합으로는 승격하지 않음.
Promoted default dual-pass

C2 Controlled Dual-Pass

복잡한 작업을 구현 전에 정규화하고, context를 좁히고, 구현 후 deterministic gate와 독립 리뷰를 반드시 거치는 구조입니다. 이 프로젝트가 현재 가장 신뢰하는 운영 형태입니다.

Triage goal, done when, non-goals, risk, invariants 정리
Context 관련 코드, 테스트, 문서만 좁혀서 주입
Execution 작은 diff, regression test, docs sync 중심으로 구현
Gates tests, lint, typecheck, docs sync, task-specific checks
Review severity 기반 독립 결함 탐지
  • 장점: unrelated change, missed test, docs drift를 줄이고 reviewer signal이 좋아집니다.
  • 약점: triage/context가 약하면 전체 파이프라인이 흔들릴 수 있고 운영 시간은 늘어납니다.
  • 최종 판정: complex-case 품질 보장용 기본 조합으로 채택.

핵심 역할 카드

현재 benchmark에 실제로 투입된 core role은 Triage Agent, Execution Agent, Review Agent입니다.

Role scope control

Triage Agent

복잡한 요청을 구현 전 단계에서 정규화해 범위 폭주와 hidden-risk omission을 줄입니다.

  • 입력: 사용자 요청, 이슈/PRD, 초기 파일 목록, 정책 제약
  • 출력: task type, goal, done when, non-goals, risk, reviewer focus
  • 금지: 구현 수행, 승인 없는 쓰기, 무차별 context packing
  • 감시 실패: wrong_context, insufficient_spec, over-scoping
Role implementation

Execution Agent

승인된 범위 안에서 구현, 테스트 보강, 문서 반영, 로컬 검증을 수행하는 실행 중심 역할입니다.

  • 입력: task brief, context pack, deterministic gates, docs sync 요구
  • 출력: 변경 요약, 변경 파일, 테스트 결과, reviewer focus points
  • 금지: 범위 밖 대규모 리팩터링, gate 실패 무시, 승인 없는 파괴적 작업
  • 감시 실패: shallow_fix, over_edit, missed_test, docs_desync
Role independent reviewer

Review Agent

요구사항 누락, 회귀 위험, 테스트 부족, 운영/보안 리스크를 severity 기반으로 독립 탐지합니다.

  • 입력: task brief, context pack, diff, gate 결과
  • 출력: High / Medium / Low / Residual Risks
  • 금지: 구현 재수행, 낙관적 요약으로 findings 대체, severity 없는 코멘트
  • 감시 실패: false reassurance, missed regression, requirements drift

Skill Atlas

실제 품질 개선의 대부분은 persona가 아니라 skill의 절차 강제력에서 나옵니다. 현재 시스템은 core 7개 + optional 1개 skill로 구성됩니다.

Wave 1 briefing

task-brief-author

ambiguous request를 strict task brief로 변환합니다. goal, done conditions, non-goals, invariants, counterexamples, approvals를 명시합니다.

  • scope creep 억제
  • collision, idempotency, stale-event, tie-case 위험을 브리프 단계에서 노출
  • 줄이는 실패: vague success criteria, hidden-risk omission
Wave 1 context

context-pack-builder

전체 저장소가 아니라 관련 코드, 테스트, 최근 변경, 문서만 선택해서 execution을 빠르고 안전하게 만듭니다.

  • 각 파일 포함 이유를 명시
  • 작고 날카로운 pack 우선
  • 줄이는 실패: broad context dump, missed regression file
Wave 1 implementation

execution-engineering

작업 invariant를 다시 확인하고, 가장 작은 coherent diff로 구현한 뒤 regression coverage와 docs sync까지 강제합니다.

  • task-scoped fix 우선, broad abstraction 억제
  • 키/순서/reconnect/telemetry touching 시 edge-case 점검 강제
  • 줄이는 실패: over-edit, shallow fix, stale/tie-case miss
Wave 1 review

review-findings

구현 재작성 대신 concrete finding만 severity별로 리포트하도록 강제합니다.

  • 요구사항 누락, 회귀, test gap, operational hazard 우선
  • finding마다 파일/행동/재현 맥락 연결
  • 줄이는 실패: generic praise, weak severity calibration
Wave 1 verification

quality-gates-runner

tests, lint, typecheck, docs sync를 고정 순서로 돌리고, task risk에 따라 migration, retry, concurrency gate를 추가합니다.

  • 필요한 gate를 생략해 pass를 만들지 않음
  • missing gate는 explicit missing으로 기록
  • 줄이는 실패: skipped gate disguised as pass
Wave 1 trace

trace-eval-recorder

candidate id, task id, model mapping, changed files, gate 결과, review severity, taxonomy tags를 구조화해 남깁니다.

  • run 하나당 trace 하나
  • blocking failure를 summary에 납작하게 눌러버리지 않음
  • 줄이는 실패: impression-driven tuning
Wave 1 doc hygiene

docs-sync

코드 변경이 사용자 행동, 운영 행동, 아키텍처 진실을 바꿨는지 판단하고 맞는 문서를 blocking update로 처리합니다.

  • runbook/ADR/operator guidance 중심
  • “테스트가 있으니 문서는 안 바꿔도 된다”를 금지
  • 줄이는 실패: docs drift, runbook omission
Optional research

research-grounded

외부 SDK, 정책, 프로토콜 같은 결정을 local repo context만으로 안전하게 내릴 수 없을 때 primary source 기반으로 조사합니다.

  • facts / inference / unresolved uncertainty 분리
  • speculative conclusion 금지
  • 줄이는 실패: hallucinated policy interpretation, version miss
P0 control-plane release

release-gate

gate 통과와 실제 release-ready를 구분해, blocker, missing approval, rollback readiness를 근거와 함께 판정합니다.

  • release-ready / hold / escalate 중 하나로 결정
  • unresolved High, skipped gate, missing rollback을 blocker로 취급
  • 줄이는 실패: false release-ready, operator handoff ambiguity
P0 control-plane approval

approval-policy

변경을 approval tier로 분류하고, 지금 허용 가능한 작업과 사람 승인 전 멈춰야 할 작업을 명확히 분리합니다.

  • read-only / safe / moderate / risky / irreversible 분류
  • small diff라도 privileged, destructive, production-affecting change를 보수적으로 분류
  • 줄이는 실패: silent risky allow, irreversible underclassification
P0 control-plane security

security-review

일반 코드 리뷰와 분리된 보안 lane에서 auth, tenant isolation, data exposure, secret handling, auditability를 검토합니다.

  • Critical / High / Medium / Low / Residual Security Risks 형식 유지
  • exploit path 또는 misuse path 중심으로 severity를 판단
  • 줄이는 실패: auth bypass miss, secret flow omission, audit gap underclassification
P1 routing ownership

ownership-routing

변경 파일과 서비스 criticality를 기반으로 owner team, required reviewers, release approvers, stricter gate lane을 계산합니다.

  • path-based deterministic routing 우선
  • unmapped critical path를 조용히 넘기지 않고 routing gap으로 남김
  • 줄이는 실패: unowned change silence, critical service under-routing
P0 governance cost

cost-governance-trace

role별 model, effort, token, latency, owner, approval, release state를 trace에 넣어 quality와 cost를 함께 비교할 수 있게 만듭니다.

  • quality-first route와 cost-optimized route를 수치로 비교
  • human intervention과 governance overhead를 숨기지 않음
  • 줄이는 실패: cost-only optimization, latency blind routing
P1 policy incident

incident-mode-policy

incident와 hotfix 상황에서 어떤 shortcut이 허용되고 어떤 통제는 절대 유지해야 하는지 명시해, 빠르게 움직이면서도 감사 가능성을 유지합니다.

  • freeze override, required communications, rollback obligations 명시
  • incident mode에서도 auth/data/irreversible change는 named approver 필요
  • 줄이는 실패: incident shortcut overreach, freeze bypass without record

모델 / Effort 해석

gpt-5.4와 gpt-5.4-mini는 “어느 역할에 배치하느냐”가 더 중요했고, xhigh는 품질의 만능 해답이 아니었습니다.

Quality-first default

Triage / Execution / Review = gpt-5.4 / high

현재 가장 안전한 기본값입니다. triage의 품질, execution의 범위 통제, review의 결함 탐지력이 모두 안정적으로 유지됐습니다.

  • refined batch에서 0 finding
  • 복잡한 ordering, cache, telemetry 문제에서 과잉 일반화가 줄어듦
  • 프로덕션 기본 승격에 적합
Efficiency probe

Triage/Context = gpt-5.4-mini / medium

triage/context는 mini downshift 가능성이 가장 크게 보이는 영역입니다. broad sweep에서 quality가 무너지지 않았고, wrong-context 계열이 급격히 늘지 않았습니다.

  • 가장 낮은 token total among robust configs
  • fastest average wall time among promoted structures
  • 다만 refined 기준의 추가 확인 batch는 아직 남아 있습니다
Important warning

execution = xhigh is not a default

execution에서 xhigh는 생각을 깊게 하는 대신, task invariant를 넘어선 state rewrite와 broad abstraction을 유발할 수 있었습니다.

  • initial batch에서는 한 번 이득이 있었지만
  • refined batch에서는 stale reconnect regression으로 High 발생
  • 결론: xhigh는 default가 아니라 예외적 실험값

Role Routing Policy

  • Triage / Context: 먼저 gpt-5.4-mini / medium 후보를 고려합니다.
  • Execution: 기본값은 gpt-5.4 / high입니다. 고위험 구조 문제에서도 이 조합이 가장 안정적이었습니다.
  • Review: complex-case에서는 gpt-5.4 / high를 유지합니다. 독립 리뷰는 절대 생략하지 않습니다.

Interpretation Rules

  • mini가 충분하다고 보려면 primary metrics가 유지되고 role-specific taxonomy가 악화되지 않아야 합니다.
  • 품질 차이가 미미한데 latency와 cost만 증가하면 over-provisioned로 봅니다.
  • skill과 role contract를 고정한 뒤 model/effort만 바꿔야 비교가 정직해집니다.
